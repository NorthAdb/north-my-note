## 四、核心概念

### 核心概念 1：推理的定义（多规则）

多规则类推理具有三个特征：**多个条件约束**、**多个相关步骤相互依赖**、**需要思考与推理才能做对**。

**参考答案**：

2. **考察模型在复杂规则下的推理**：模型需在规划时考虑所有约束，且每个约束是**双向选择**（学员需满足导师要求，导师也要满足学员）。模型需时刻关注规则，进行试错，得出正确答案

2. **定义了一套相互依赖的复杂规则**：
   - **条件依赖与顺序执行**：必须严格遵循 A→B→C→D 的顺序，A 和 C 相互依赖（A 依赖 C 的初始空间，C 依赖 A 的最终空间）
   - **优先级与保护机制**：核心数据永远不删除，用户缓存数据可被牺牲，独立分区（B、D）只能本地删除

### 核心概念 2：规划的定义（Planning）

> **推理与规划的区别**：推理不一定能是规划，但规划一定涉及推理。
> - **推理**：从已知→找结论（条件、推演）
> - **规划**：从目标→找行动（思考、决策、反思、执行）
>
> 如果无法严格判断题目类型，可以从**考点**出发：题目核心目的考**逻辑推导**则选推理，考**执行方案**则选规划。

### 核心概念 3：系统提示词（SP）与用户提示词（UP）定义

| 类型 | 说明 |
|------|------|
| **系统提示词（SP）** | 定义模型角色、行为规范、输出格式等全局性指令 |
| **用户提示词（UP）** | 具体的任务描述、输入数据、问题场景等 |

### 核心概念 4：文本自然度

| 特征 | 解释 | 示例/对比 |
|------|------|-----------|
| **语言符合真实使用场景** | 使用自然口语或业务场景中的真实表达 | ✅「现在有3台无人机需要分配任务，每台每天只能执行一次飞行。」❌「在当前情境设定下，存在三个无人机实体，其执行行为需遵循既定操作约束。」|
| **信息呈现有重点** | 人类写作会强调关键限制，次要信息较简略 | AI文本常见问题：每条规则长度相似，所有内容重要性看起来一致 |

## 五、作业流程

### 2.0 出题背景及类型

#### 2.0.1 撰写出题背景

目的：帮助初读者快速了解题目在考什么。好的出题背景会说明「出题的场景、它的核心考点是什么」，一般控制在 **50 字内**。

![[feishu-mianbi/workflow-2.0.1-background-example.png]]

#### 2.0.2 选择题目类型并检测配额

**必选项**。【复杂推理】和【复杂任务规划】各占总量的 1/2。选择类型后需点击【检查配额】：

![[feishu-mianbi/workflow-2.0.2-quota-check.png]]

- ✅ **通过** → 该类型量级可继续出题
- ❌ **不通过** → 该类型已达上限，只能选另一类型
- 两个类型均满 → 项目进入结项阶段

> 一般的规划类任务都具备推理属性，需要从任务目标进行区分。

### 2.1 撰写系统提示词（SP）

**SP 不得少于 100 字符**。核心要求：

| 要求 | 说明 |
|------|------|
| **内容结构化** | 按"分类→分层"撰写，用有序列表或编号等可视化形式呈现 |
| **描述无歧义**（⭐️重点检查） | 表述明确、清晰，明确逻辑关系，无暗示、歧义词汇，前后规则/指令表达一致 |
| **逻辑步骤闭环** | 规则和指令明确：允许行为、禁止行为、特殊场景处理办法，不存在灰色地带或规则耦合导致暗示、冲突 |
| **条件边界清晰** | 每个规划类题目至少包含 3 个及以上条件、规则（推理题至少 3 个以上条件），这些条件是作用于任务的前提条件，不能是纯基础指令 |
| **不联网即可作答** | 题目只需要依据题目的规则/指令即可完成任务，无需联网 |
| **格式标准化**（⭐️） | 题目中一定要明确最终答案的输出结构（如某个数值、一段编码、JSON、Markdown 列表等）。但不要约束"不要输出过程"，只需规定好输出格式即可 |
| **容错兜底** | 在信息缺失、极端场景处给出明确的兜底处理方式，避免模型猜测执行 |

![[feishu-mianbi/workflow-2.1-SP-example.png]]

### 2.2 撰写用户提示词（UP）

在用户提示词栏撰写 UP，**无长度限制**。不要在 SP 和 UP 撰写栏中提及"上文 SP"或"下文 UP"这类词汇。

核心要求：**任务描述清晰、具体、客观，答案有收敛（唯一解或有限个解），不存在开放性疑问**。

| 要求                       | 说明                           |
| ------------------------ | ---------------------------- |
| **确保任务清晰具体**             | 任务明确具体，无多义，避免存在阐述模糊、前后矛盾、错别字 |
| **确保得出的结果收敛、可观、可评估**（⭐️） | 确保问题的结果是收敛到唯一解或有限个解，避免结果主观   |

![[feishu-mianbi/workflow-2.2-UP-example.png]]

### 2.4 测试题目难度（难度要求）

> 🎯 本任务的挑战是要出一道"可以难倒"目标模型的题目（模型必须做错），所以一个潜在的风险是【专家花了很多时间写{题目}+{思考过程}+{参考答案}】但是【题目太简单，导致前面的信息全部要重写】，导致时间的浪费。

**初步检测**：作业模板中插入了【初步难度检测】（模型0），用户辅助判断难度：
- **情况1**：模型0做错了 → 难度可能达标，继续撰写{思考过程}与{参考答案}
- **情况2**：模型0做对了 → 难度不达标，建议先修改题目再继续

**后续模型得分要求**：模型一和模型二均无法做对（无法做对主要指模型在过程中的核心步骤或结果存在错误，而非一些表述瑕疵或未要求要求的格式、风格、语气等问题）。模型三、四属于强力模型，其回答正确性不影响题目提交。

![[feishu-mianbi/workflow-2.4-difficulty-test.png]]

### 2.5 撰写解题过程（trajectory）

当题目质量检查与模型测试后发现无需进行题目修改后，即可撰写解题过程。核心要求：**逻辑清晰、循序渐近**。

- trajectory 必须是一个**正向、循序渐近**的步骤
- 每一步都是具体的、可执行的，并且是前一步的逻辑延续，**不能跳步**（当做在教一个小学生做题）
- 如使用了真实世界中的定义、公理、公式等，需要前置说明定理名称
- trajectory 的最后部分需要给出题目的**最终结果**，不能省略
- 对于多答案的 trajectory，需要至少给到一个答案的完整解题路径，其余答案可以在最后用`<其他解题过程说明>`简述

| 合格标准 | 不合格标准 |
|---------|-----------|
| 步骤完整、逻辑严密、上下文因果关系紧密 | trajectory推导过程出错（出现漏洞、歧义、前后矛盾） |
| 格式正确，可读性高（步骤清晰，有序/无序列表穿插使用） | trajectory存在明显跳步（缺少可解释性） |
| 数理化信息使用 LaTeX 语法表达（含表格） | trajectory存在渲染问题（需使用 LaTeX 且能被 StackEdit 完整渲染） |
| 多答案时至少给到一个 answer 的完整解题过程 | trajectory存在题目外信息（出现"UP""SP""模型0"等题目内不存在的流程上字眼） |
| trajectory最后给出最终结果 | trajectory缺失了最终结果 |

### 2.6 撰写答案（answer）

需要撰写题目的最终答案，注意格式需要与前面 SP 或 UP 中规定的格式一致。

- answer **不包含解题过程信息**，只需给出一个正确、完整、收敛的最终结果
- 如果 answer 表达过长，建议修改 prompt 形式使其成为一个精简解（如"它一共需要调用多少个工具？""该题共多少解？""给出最终信息无需额外注释"等）

| 合格标准 | 不合格标准 |
|---------|-----------|
| 答案正确、收敛且已被穷举 | 答案错误（最终结果有误，与 Prompt 不匹配） |
| 格式正确（遵循 Prompt 规定的输出格式） | 格式违背 prompt 规定 |
| 多答案时全部已被穷举 | 存在其他结果、答案未被穷举 |
| 数理化信息使用 LaTeX 语法 | answer正确性无问题但表述过长 |

### 2.7 测试回复并打分备注（评分）

> 🎯 线上内置了四个匿名模型：模型一、二为一组；模型三、四为一组，需要进行模型回复测试，并对回复结果进行打分。

**打分规则**：只有"错误(0)"和"正确(1)"两档。对模型的过程和结果进行打分——过程和结果任一个存在关键步骤（解题核心步骤）或结果错误，就视为"错误(0)"。当模型回复的结果正确或同义（答案形式不同但表达含义一致）即可给"正确"；反之答案错误、不全则给"错误"。

**打分备注**：仅对回复错误的模型回复进行备注，用 1.2.3.…列举其错误的原因；对回复正确的结果无需备注。

### 2.8 AI 质量检查

为节约大家的检查时间，作业关键检验节点内置了 AI 查错，主要环节有：

1. **prompt（SP+UP）**：AI 生成痕迹与流畅度检查
2. **trajectory**：逻辑步骤正确性与跳步检查
3. **题目测试打分**：正确性检查
4. **answer**：正确性检查

> ⚠️ 每次修改质检内容后，都应点击重新检测。如不认可 AI 检测结果可以备注一下不认可的理由（可以接着进行下一步），后续会仔细阅读理由进行复核。

## 八、反例积累

项目团队会对作业中出现的 badcase 进行提炼，供专家每日查看以规避常见错误。

### 反例 1：纯指令类创作题，无收敛结果

**问题**：题目为开放性创作，没有标准答案/收敛结果，模型输出无法客观评判。

**题目示例（截取）**：
> 请按以下要求为辽宁省设计一个2025年营商环境第三方评估方案。
> 1. 标题中体现地区和年份。
> 2. 方案涵盖评估背景、评估目的与意义、评估原则、评估体系...
> 3. 评估背景要体现习近平总书记关于东北地区营商环境的指示要求...
> ...
> 10. 要明确评估需要完成的最终成果，以及评估成果如何应用。

**问题分析**：虽然指令详细（10 条要求），但本质是「创作题」——没有封闭式答案、没有唯一收敛结果。这类题目无法作为模型评测数据使用。

### 反例 2：最优化问题，参考答案并非最优

**问题**：题目是最优化类问题，但给出的参考答案并非真正最优解，导致评测失效。

**题目背景**：
> 医院手术室调度问题（RCPSP 变体），包含手术室分配、手术排序和时间安排三个决策维度。12 台手术、4 间手术室、6 位医生。约束包括时间窗、资源冲突、手术类型互斥、顺序限制等。优化目标：最小化完工时间和加权等待时间。

**错误原因**：给出的「最优排程」方案实际上不是最优解，意味着即使是正确答案也无法被正确评判。
