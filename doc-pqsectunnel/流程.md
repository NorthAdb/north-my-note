## 讲演稿：PQ-SecTunnel 握手流程串讲

各位老师好。我按这张流程图，把双方如何完成后量子握手、最终得到会话密钥说清楚。

### 一、开场：我们在解决什么

经典 WireGuard 用 Noise IK，密钥交换依赖 ECDH。我们把 DH 换成 KEM，协议形态仍接近 IK，但变成 1.5-RTT、三条消息：Initiation、Response、Key Confirmation。密码学上双方各自维护两条状态链：哈希链 H（累计 transcript，当 AEAD 的附加认证数据）和 链式密钥 C（不断 KDF 混入密钥材料，最后派生会话密钥）。图里主要画了哈希链的初始化与更新；链式密钥是同步演进的，图上没单独画出。

---

### 二、预配置与证书加载

握手开始前，双方必须已经知道对方的长期静态公钥。

- 默认：配置文件里写好 peer 的静态公钥。
- 也可走 PKI：带外验证证书后，从证书取出对端长期公钥。
- 可选还有预共享密钥 PSK；未配置时用本地规则派生一个默认值。

说明：静态公钥用于身份认证和查 peer；临时密钥不在预配置里，每次握手由内核现生成。Cryptokey Routing 里的 AllowedIPs 也是预配置的，主要用于握手成功后的数据面选路，不是这三条握手消息本身的内容。

---

### 三、消息 1：Handshake Initiation（发起方 → 响应方）

发起方要做四件事，然后组包发出。

第一，生成临时 PQ 密钥对（主套件是 ML-KEM-768）。公钥 `E_i^pub` 明文放进报文；私钥只留在发起方内存，握手后可销毁。这是前向安全的关键。

第二，初始化哈希链（以及链式密钥）：从协议名标签出发，再混入响应方长期公钥 `S_r^pub`、以及刚生成的 `E_i^pub`，双方对“协议身份、我对谁说话、本轮临时公钥是什么”对齐。

第三，对响应方长期公钥做一次 KEM 封装，得到 `ct₁` 和共享秘密 `shk₁`：  
`(ct₁, shk₁) = KEM.Enc(S_r^pub)`。  
说明：这对应 Noise 里用临时/静态材料认证响应方身份的那一步——只有真正持有 `S_r` 私钥的人才能解封装，从而继续后面的 AEAD。图中绿框「KEM密文」在消息 1 里主要就是 `ct₁`。

第四，用 AEAD 保护身份相关字段：派生出阶段性密钥后，加密发起方静态身份摘要（图中「加密的身份信息」）和时间戳，并用当前哈希值作 AAD，保证字段与 transcript 绑定。

报文里还可以看到发起方会话号 `sid_i`，以及 mac1/mac2。说明：mac 不进 C/H 链，是 WireGuard 式的入口防伪/DoS 保护；接收端先验 MAC，再进握手状态机。

响应方收到后：验 MAC 与时间窗口 → 用自己的静态私钥解 `ct₁` → 更新自己的 H/C（与发起方对称）→ AEAD 解密得到发起方身份 → 在 peer 表里认出是谁。到这里，响应方已经确认：“有人拿着某个临时公钥在对我的长期身份做封装，并且身份密文能解得开。”

若主套件报文超过 MTU，消息 1/2 会在应用层切成多个 UDP 发出，对端按分片头重组后再走上述逻辑；MAC 是对完整握手算完再切的。

---

### 四、消息 2：Handshake Response（响应方 → 发起方）

响应方在已对齐的 transcript 上继续更新哈希链和链式密钥，然后做两次 KEM 封装——这是本协议相对经典 IK 最醒目的地方：

1. `ct₂ = Enc(E_i^pub)`（对消息 1 带来的临时公钥）  
    得到本轮临时共享秘密。图中「PQ临时KEM密文」。目的是双方共建新鲜秘密、提供前向安全；封装方注入随机性，解封装方持有临时私钥。
    
2. `ct₃ = Enc(S_i^pub)`（对发起方长期公钥）  
    图中「PQ长期KEM密文」。只有真正的发起方能解，用来认证发起方身份（对应 Noise 里 se 一类步骤）。
    

然后用新派生的阶段性密钥 AEAD 加密一个空明文（empty），作为响应方向发起方的密钥确认材料——证明响应方已走到同一密钥阶梯。图中写成「加密的验证值」，指的就是这个 authenticated empty，而不是又传一份会话密钥。

报文带上响应方会话号 `sid_r`、回引 `sid_i`，再加 MAC。

发起方收到后：验 MAC → 用 `E_i^priv` 解 `ct₂`、用自己的静态私钥解 `ct₃` → 更新 H/C → AEAD 解开 empty。若成功，发起方已经掌握派生会话密钥所需的全部材料，链式密钥可走到 `C₁₀`，并抽出确认用的 `K₄`。

说明：会话传输密钥 `TK_i`/`TK_r` 并不在网上传递，而是双方在本地对 `C₁₀` 做 KDF；消息 2 只交换密文和确认密文，让两边状态对齐。

---

### 五、消息 3：Key Confirmation（发起方 → 响应方）

为什么还要第三条？经典 WireGuard 往往靠第一条加密数据包隐式确认；后量子报文更大、握手更重，我们把确认做成显式第三条消息，整次握手是 1.5-RTT。

发起方再更新哈希链，用 `K₄` AEAD 加密确认值（同样可以是空明文 `conf`），证明：“我已成功处理消息 2，双方密钥一致。” 带上双方会话号和 MAC。

响应方：验 MAC → AEAD 解密确认值。成功则同样从 `C₁₀` 派生 `TK_i`/`TK_r`，安装会话密钥对，握手完成，进入数据面。

说明：消息 3 不再做 KEM，不再贡献新的共享秘密；只做密钥一致性证明。之后数据面用 SM4-GCM 等对称算法加密整条内层 IP 包；出站按目的 IP 查 AllowedIPs 选 peer，入站解密后再核源 IP 是否落在该 peer 的 AllowedIPs。

---

### 六、收束：三条消息各自角色（答辩可背）

|消息|核心密码动作|一句话目的|
|---|---|---|
|Initiation|Gen 临时钥；`Enc(S_r)`；AEAD 身份与时间|声明本轮临时身份，并开始认证响应方|
|Response|`Enc(E_i)` + `Enc(S_i)`；AEAD empty|注入临时秘密与认证发起方，并证明响应方密钥已就绪|
|Key Confirm|仅 AEAD conf|发起方显式确认密钥一致，对称开启会话|

三次 KEM 可记成：  
`ct₁` 认证响应方 · `ct₂` 临时前向安全 · `ct₃` 认证发起方。  
哈希链管“双方看到的内容是否一致”，链式密钥管“最终对称密钥从哪长出来”；MAC 在入口挡篡改，不参与密钥衍生。
