# 07 — 工程架构与实验验证（完整流程）

> 状态：正文 | 主套件：`mlkem512-mlkem768-sm4gcm-sm3`  
> 预计篇幅：答辩 5–8 min（可拆成「架构 2 min + 实验演示 5 min」）

---

## 0. 先建立直觉：这个项目分两层世界

想象 PQ-SecTunnel 是一台「加密路由器」：

| 世界 | 类比 | 在本项目里是什么 |
|------|------|-----------------|
| **用户态** | 管理员用遥控器配网 | `pqst`、`pqst-quick`：写配置、生成密钥、告诉内核「跟谁连」 |
| **内核态** | 路由器芯片真正转发加密包 | `pqsectunnel.ko`：握手、KEM/AEAD、分片、UDP 收发 |

**关键一点**：密钥和 peer 信息通过 **Netlink** 从用户态下发；但握手和密码运算**全部在内核里跑**，用户态不参与运行时 KEM（CONTEXT **pqst** / **pqsectunnel.ko**）。

本套件是 **编译时** 定死的算法组合（512 静态 + 768 临时 + SM4-GCM + SM3），**不能**在运行时切换。

## 2. 阶段 A：构建（`build.sh`）— 把源码变成可运行套件

### 2.1 你要执行的命令

```bash
cd /path/to/PQ-SecTunnel
sudo ./build.sh -e mlkem512-mlkem768-sm4gcm-sm3 --keep-experiments
```

| 参数 | 作用 |
|------|------|
| `-e <名>` | 只构建这一个实验，不编译其它套件 |
| `--keep-experiments` | 不删掉 `experiments/` 里其它已构建套件（推荐） |

**需要 root**：内核模块编译/加载环节脚本会检查权限。

### 2.2 构建内部做了什么（按顺序）

#### 步骤 A1：读「配方单」

读取 `wgtest/wgtest-mlkem512-mlkem768-sm4gcm-sm3/vars`：

```bash
STATIC_KEM=mlkem512
EPHEMERAL_KEM=mlkem768
AEAD=SM4GCM
HASH=SM3
NOISE_HANDSHAKE_NAME=Noise_IKpsk1kem_SM4GCM_SM3_EphemMLKEM768
```

**作用**：决定编译进 `.ko` 的算法；相当于选定「这盒 VPN 芯片的硬件规格」。

#### 步骤 A2：编译内核模块 `pqsectunnel.ko`

| 动作 | 产出 | 作用 |
|------|------|------|
| `kem` → symlink 到 `kems/mlkem-512` | — | Static KEM 用 512 实现 |
| 注入宏 `-DEPHEMERAL_KEM_MLKEM768` 等 | — | Ephemeral 用 768；Hash=SM3；AEAD=SM4-GCM |
| 检测 Initiation struct >1400B | `-DPQST_APP_LAYER_FRAGMENTATION` | 本套件**启用**应用层分片 |
| `make` in `wireguard-linux-compat/src` | `pqsectunnel.ko` | 拷贝到 `experiments/.../` |

**小白理解**：`.ko` 是 Linux **内核驱动**，加载后内核才具备 PQ 握手能力。里面的 KEM/AEAD 在编译时已焊死，不能换。

#### 步骤 A3：编译用户态工具

| 动作 | 产出 | 作用 |
|------|------|------|
| `make nist-mlkem512-genkey` | `nist-mlkem512-genkey` | 按 NIST 标准生成 800B/1632B 的 ML-KEM-512 密钥对 |
| `make pqst` | `pqst` | 配置接口：`genkey`、`set`、`show` 等 |
| 复制 `wg-quick/linux.bash` | `pqst-quick` | 读 `.conf` 拉起虚拟网卡 |

**小白理解**：`pqst` 是「管理员 CLI」；`pqst-quick` 是「读配置文件一键启动」。

#### 步骤 A4：复制实验脚本 + 写 `suite_meta.env`

从 `resources/` 复制 `run.sh`、`setup_ns.sh`、`setup_configs.py` 等到实验目录。

生成 `suite_meta.env`（构建身份证），本套件示例：

```
STATIC_KEM=mlkem512
EPHEMERAL_KEM=mlkem768
APP_LAYER_FRAGMENTATION=auto
APP_LAYER_FRAGMENTATION_EFFECTIVE=1
NOISE_HANDSHAKE_NAME=Noise_IKpsk1kem_SM4GCM_SM3_EphemMLKEM768
```

**作用**：后续 `setup_configs.py` 读它校验密钥长度；答辩时可证明「跑的就是这套算法」。

### 2.3 阶段 A 完成后你拥有什么

- 一个**自包含**目录，可拷到测试机独立运行
- **尚未**生成 `.conf`、**尚未**加载模块、**尚未**握手
- 相当于「工具箱准备好了，还没装机组网」

## 3. 阶段 B：运行实验（`run.sh`）— 真连线 + 采日志

### 3.1 你要执行的命令

```bash
cd experiments/mlkem512-mlkem768-sm4gcm-sm3
sudo ./run.sh                    # 基础：握手 + 采 dmesg
sudo ./run.sh --with-transfer      # 握手 + iperf3 吞吐测试
```

**必须在实验目录内执行**——脚本依赖同目录的 `.ko`、`pqst`、`setup_configs.py`。

### 3.2 实验网络长什么样（两个「平行宇宙」）

`setup_ns.sh` 用 **network namespace（netns）** 模拟两台机器：

```
┌──────────────────────── cli0_ns（客户端 / Initiator）────────────────────────┐
│  虚拟网卡 cli0_ve  IP: 10.20.0.2/24                                          │
│  VPN 接口 pqst0    IP: 10.0.8.1/24   ← 隧道「内网」地址                        │
│  pqst-quick up pqst0.conf                                                    │
└───────────────────────────────┬──────────────────────────────────────────────┘
                                │ veth 对 + tc netem（默认 30ms 延迟）
┌───────────────────────────────┴──────────────────────────────────────────────┐
│  虚拟网卡 srv0_ve  IP: 10.20.0.1/24                                          │
│  VPN 接口 pqst0    IP: 10.0.8.2/24                                           │
│  pqst-quick up server0.pqst0.conf                                            │
└──────────────────────── srv0_ns（服务端 / Responder）────────────────────────┘
```

| IP 段 | 含义 |
|-------|------|
| `10.20.0.x` | **Underlay**：模拟公网/链路层，UDP 握手走这里（51820 端口） |
| `10.0.8.x` | **Overlay**：VPN 隧道内网，ping/iperf3 走这里（握手成功后通） |

**小白理解**：netns 像两台独立 Linux 虚拟机在同一台物理机上；veth 是它们之间的网线；netem 在网线上加延迟，模拟真实 WAN。

### 3.3 `run.sh` 逐步详解（8 步）

下面每一步对应 `run.sh` 源码顺序。

---

#### 步骤 B1：`reset.sh` — 清理旧环境

```bash
sudo ./reset.sh
```

| 产出 | 作用 |
|------|------|
| 删除所有 `ip netns` | 上次实验残留的 `cli0_ns`、`srv0_ns` 清掉，避免 IP/接口冲突 |

**小白理解**：实验前先「恢复出厂设置」。

---

#### 步骤 B2：`rmmod pqsectunnel` — 卸载旧内核模块

```bash
sudo rmmod pqsectunnel || true
```

| 产出 | 作用 |
|------|------|
| 内核卸载旧 `.ko` | 确保接下来加载的是**本目录新编译**的模块，不是系统里别的版本 |

---

#### 步骤 B3：`setup_configs.py` — 生成密钥和配置文件

```bash
python3 ./setup_configs.py
```

**内部流程**：

1. 读 `suite_meta.env`，确认 Static KEM = mlkem512
2. 调用 `./pqst genkey`（默认 `PQST_KEY_SOURCE=random`）生成两对 ML-KEM 静态密钥：
   - Initiator：`InitiatorPublicKey` / `InitiatorPrivateKey`
   - Responder：`ResponderPublicKey` / `ResponderPrivateKey`
3. 校验 Base64 解码后公钥 800B、私钥 1632B（512 标准）
4. 写入两个配置文件

| 产出文件 | 角色 | 主要内容 |
|---------|------|---------|
| **`pqst0.conf`** | 客户端 Initiator | 本机 Initiator 私钥/公钥；Peer 填 Responder 公钥；`Endpoint = 10.20.0.1:51820` |
| **`server0.pqst0.conf`** | 服务端 Responder | 本机 Responder 私钥/公钥；Peer 填 Initiator 公钥；`ListenPort = 51820` |

**作用**：把「谁是谁、密钥是什么、连哪里」写成 WireGuard 风格配置；**Ephemeral 密钥不在 conf 里**——每次握手由内核临时生成 E_i（ML-KEM-768）。

**小白理解**：`.conf` 是两张「身份证 + 通讯录」，还没有真正建隧道。

可选：`PQST_KEY_SOURCE=pki` 时从证书库读 Static 公钥（见 [08](08-PKI与密钥管理.md)）。

---

#### 步骤 B4：`dmesg -c` — 清空内核日志缓冲区

```bash
sudo dmesg -c >/dev/null
```

| 产出 | 作用 |
|------|------|
| 空 dmesg  ring buffer | 后面采集的 `dmesg.log` **只含本次实验**，便于答辩展示握手/分片日志 |

---

#### 步骤 B5：`insmod pqsectunnel.ko` — 加载内核模块

```bash
sudo modprobe udp_tunnel
sudo modprobe ip6_udp_tunnel
sudo insmod pqsectunnel.ko
```

| 产出 | 作用 |
|------|------|
| 内核注册 `pqsectunnel` 模块 | 出现 `/dev` 网卡能力；`wg_noise_init()` 用 **SM3** 预计算 Noise 协议标签 |
| dmesg 出现 `PQWireGuard 2.0.0 loaded` | 证明模块加载成功 |

**小白理解**：把「加密 VPN 引擎」插进内核；没这步，pqst-quick 配置了也加不了密。

---

#### 步骤 B6：`setup_ns.sh` — 建 netns + 启动隧道（**握手发生在这里**）

```bash
sudo ./setup_ns.sh 0 1 &
# 若有多个 pqst*.conf，run.sh 会对每个 index 并行调用
```

**子步骤 6a — 创建隔离网络**

| 动作 | 产出 |
|------|------|
| `ip netns add cli0_ns` / `srv0_ns` | 两个独立网络栈 |
| `ip link add ... type veth` | 一对虚拟网线连接两 netns |
| 分配 `10.20.0.2` / `10.20.0.1` | Underlay IP |
| `tc netem delay 30ms rate 1000mbps` | 链路仿真（可调 `NETEM_DELAY_MS`） |

**子步骤 6b — 启动 VPN 接口**

```bash
ip netns exec srv0_ns ./pqst-quick up ./server0.pqst0.conf
ip netns exec cli0_ns ./pqst-quick up ./pqst0.conf
```

| 动作 | 作用 |
|------|------|
| `pqst-quick` 解析 `.conf` | 通过 Netlink 把密钥、peer、AllowedIPs 下发内核 |
| 创建 `pqst0` 虚拟 TUN 设备 | 客户端得 `10.0.8.1`，服务端得 `10.0.8.2` |
| 客户端主动发 **Handshake Initiation** | 内核 `send.c` → 2068B → **应用层分 2 片** |

**子步骤 6c — 内核里自动完成的握手（你看不见，但在 dmesg 里）**

1. Responder 收 Initiation → 重组 2068B → 验 MAC → `noise.c` 解包 → 发 Response（1916B，2 片）
2. Initiator 收 Response → 重组 → 发 Key Confirmation（60B，单片）
3. 双方 `begin_session` → 派生 **TK_i / TK_r**（SM4-GCM 会话密钥）

详见 [03](03-握手协议逐步详解.md)、[05](05-内核实现路径.md)、[06](06-分片与MTU.md)。

**子步骤 6d — 等待握手成功**

```bash
# setup_ns.sh 内：最多 30 秒，每 2 秒 ping 一次 VPN 内网
ping 10.0.8.2   # 从 cli0_ns 发起
```

| 现象 | 含义 |
|------|------|
| ping 通 | 握手 + 数据面 OK |
| 输出 `Handshake confirmed after Ns` | 实验脚本判定成功 |

**本套件 dmesg 关键行**（`dmesg.log`）：

```
pqc_fragment: fragmented into 2 parts, total_len=2068      # Initiation 发送
pqc_fragment: reassembled skb->len=2068                    # Responder 重组
server to [...] completed handshake in 980087 ns           # Response 完成
pqc_fragment: fragmented into 2 parts, total_len=1916      # Response 发送
client [...] completed handshake in 80600576 ns            # Initiator 收 Response
Receiving handshake key confirmation                       # 会话建立
```

---

#### 步骤 B7：`sleep 10` — 等待所有并行实例握手稳定

多实例时 `run.sh` 会对每个 `pqstN.conf` 后台跑 `setup_ns.sh`，主流程 `wait` 后 sleep，避免过早采日志。

---

#### 步骤 B8（可选）：`test_transfer.sh` — 数据面验证

仅当 `sudo ./run.sh --with-transfer`：

| 模式 | 做什么 | 验证什么 |
|------|--------|---------|
| **iperf3**（默认） | srv 起 `iperf3 -s`；cli 跑 `iperf3 -c 10.0.8.2` | TCP 经 VPN 的吞吐量 |
| **netcat** | cli 发字符串；srv 比对收到内容 | 明文是否正确穿过 SM4-GCM 隧道 |
| **ping** | cli 对 `10.0.8.2` 发 N 个 ICMP | 延迟/丢包 |

**小白理解**：握手只证明「密钥对齐」；transfer 证明「真的能传业务数据」。

| 产出 | 作用 |

| **`dmesg.log`** | 答辩证据：模块加载、分片、握手耗时、peer 创建 |

## 8. 常见问题（小白版）

**Q：为什么 build 和 run 分开？**  
A：build 一次可复用；run 每次重新 genkey、重新 conf、重新握手，保证实验可重复。

**Q：`.conf` 里的密钥和握手里的 E_i 什么关系？**  
A：`.conf` 只有 **Static KEM** 长期钥；**Ephemeral E_i** 每次握手内核临时生成（768），不在配置文件里。

**Q：为什么 Initiation 要分片？**  
A：2068 字节 > 典型 MTU；见 [06](06-分片与MTU.md)。

**Q：ping 10.0.8.2 通说明什么？**  
A：三次握手完成 + SM4-GCM 数据面能加密 ICMP，不是只完成了前两包。

**Q：必须用 netns 吗？**  
A：实验脚本用 netns 在一台机器模拟两端；生产环境是两台真机各跑 pqst-quick。
