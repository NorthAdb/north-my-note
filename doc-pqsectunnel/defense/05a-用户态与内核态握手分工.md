# 05a — 用户态与内核态握手分工

> 状态：正文 | 关联：[05 内核实现路径](05-内核实现路径.md)、[07 工程架构](07-工程架构与实验验证.md)、[KE.md](../../KE.md)
> 主套件：`mlkem512-mlkem768-sm4gcm-sm3` | 预计篇幅：3–5 min（可答「配置怎么进内核、握手谁在做」）

## 1. 一句话定位


| 角色   | **用户态**（`pqst` / `pqst-quick`）                 | **内核态**（`pqsectunnel.ko`）               |
| ---- | ---------------------------------------------- | --------------------------------------- |
| 做什么  | 生成长期钥、读 conf、经 Netlink **下发配置**；起接口；偶尔 **查状态** | 存身份与 peer、**组/收握手包**、三次 KEM、派会话密钥、加解密数据 |
| 不做什么 | 不碰握手报文、不做 KEM                                  | 不替你改 conf、不替你生成长期钥                      |


**握手密码学与报文路径全程在内核；用户态只负责「配好再走开」。**

通道：


| 阶段                                  | 工具                                                                    |
| ----------------------------------- | --------------------------------------------------------------------- |
| 配置下发 / 状态查询                         | **Generic Netlink**（family `pqsectunnel`，`SET_DEVICE` / `GET_DEVICE`） |
| Initiation / Response / Key Confirm | **对端之间 UDP**（不经用户态）                                                   |


---



## 2. 时间线：从配钥到会话建立



### 阶段 0 — 配钥（仅用户态，一次）

`pqst genkey`（或对应 ML-KEM genkey 工具）在用户态生成静态 KEM 密钥对，写入 conf：

- **客户端**：本机 `InitiatorPrivateKey` / `InitiatorPublicKey`；对端 `ResponderPublicKey` + `Endpoint`
- **服务端**：本机 `ResponderPrivateKey` / `ResponderPublicKey`；对端 `InitiatorPublicKey`

此时尚无跨机握手；公钥经线下/工具写入对方 conf。
**用户态↔内核：无拷贝。**

---



### 阶段 1 — 配置下发（用户态 → 内核）

`pqst-quick up` / `pqst set`：

```
用户态读 conf
    │
    │  Generic Netlink：PQST_CMD_SET_DEVICE
    ▼
内核 netlink.c
    │  写入 static_identity、peer、ListenPort、socket
    ▼
接口就绪，等待握手触发
```

**用户态**：解析 conf，组 Netlink 消息。
**内核**：落地配置，为握手准备材料。

#### 拷进内核的内容（握手相关）


| 层级     | 内容                             | 作用                       | 拷贝方式                                                   |
| ------ | ------------------------------ | ------------------------ | ------------------------------------------------------ |
| Device | 本机 Initiator/Responder **公私钥** | 解 `ct_1`/`ct_3`、做本地身份    | 小钥：NLA **内联**；大钥（McEliece）：用户态指针 + 内核 `copy_from_user` |
| Device | ListenPort                     | 收握手 UDP                  | 普通 Netlink 属性                                          |
| Peer   | 对端 Responder/Initiator **公钥**  | 封 `ct_1`/`ct_3`、查 peer 表 | 同上：内联 或 PTR+`copy_from_user`                           |
| Peer   | PSK（可选）                        | 混入 KDF；未配则内核默认派生         | 内联 NLA                                                 |
| Peer   | Endpoint                       | 握手包发往哪里                  | 内联                                                     |
| Peer   | AllowedIPs / keepalive         | 触发数据面 / 保活，从而触发握手        | 内联                                                     |


判据（实现）：static 公私钥合计 **≤ 约 3200B** → 内联；更大（典型 McEliece 公钥 ~524KB）→ **PTR +** `copy_from_user`，查询时辅以 **PREVIEW**（公钥首尾片段）。

此阶段**不会**出现：临时密钥 `E_i`、会话密钥、`ct_1/ct_2/ct_3`。

---



### 阶段 2 — 触发握手（用户态间接 → 内核自行启动）

用户态在接口上发业务包 / `ping`，或配置了 PersistentKeepalive。
内核发现：有待发流量但尚无有效 keypair → `wg_packet_send_queued_handshake_initiation()`。


| 用户态             | 内核       |
| --------------- | -------- |
| 只提供「有流量要走隧道」的动机 | 决定现在必须握手 |


**此阶段双方尚未交换握手包；用户态与内核之间也不再为握手本身做钥拷贝。**

---



### 阶段 3 — Handshake Initiation（内核 ↔ 内核，UDP）

**Initiator 内核：**

1. 生成临时对 `E_i`（**仅内核**，不经用户态）
2. `ct_1 = Enc(S_r^{pub})` —— `S_r^{pub}` 来自阶段 1 拷入的对端公钥
3. AEAD 加密 `Hash(S_i^{pub})`、时间戳；算 MAC1/MAC2
4. UDP 发往 Endpoint（超 MTU 时应用层分片，仍在内核 `pqc_fragment`）

**线上交换：** `sid_i, E_i^{pub}, ct_1, static 密文, time 密文, mac1, mac2`

**Responder 内核：**

1. UDP 收包 → 先验 MAC → `noise.c`
2. 用本机 `S_r^{priv}`（阶段 1）解 `ct_1`
3. 解密得 `Hash(S_i^{pub})`，查 peer → 命中 Initiator
4. 校验时间戳 / 重放

**用户态全程不动。用户态↔内核：无握手字段拷贝。**

---



### 阶段 4 — Handshake Response（内核 ↔ 内核，UDP）

**Responder 内核：**

1. `ct_2 = Enc(E_i^{pub})` —— 绑定本次 Initiation（会话 / 前向保密）
2. `ct_3 = Enc(S_i^{pub})` —— `S_i^{pub}` 来自阶段 1 的对端 Initiator 公钥
3. AEAD 空载荷 `empty`；算 MAC；UDP 回传

**线上交换：** `sid_r, sid_i, ct_2, ct_3, empty, mac1, mac2`

**Initiator 内核：** 用本轮 `E_i^{priv}` 解 `ct_2`；用本机 `S_i^{priv}` 解 `ct_3`。

仍与用户态无关。

---



### 阶段 5 — Key Confirmation（内核 ↔ 内核，UDP）

**Initiator 内核**：用派生密钥加密空 `conf`，UDP 发出。
**Responder 内核**：解密成功 → 双方 `KDF(C_{10})` 得到 `TK_i / TK_r`，装入 keypair。

会话建立；密钥与握手状态**只在内核**。

---



### 阶段 6 — 之后（可选查询）

数据面在内核加解密；用户态经 `pqst0` 看到隧道明文。

`pqst show`（Netlink `GET_DEVICE`）可读：端口、peer 公钥（大钥或仅 PREVIEW）、LastHandshakeTime、收发字节等。

**不下发**会话密钥、`E_i`、`ct_`*。

---



## 3. 总览图

```
[用户态] genkey → 写 conf
     │
     │ Netlink SET：本机公私钥、对端公钥、PSK、Endpoint、AllowedIPs
     │ （小钥内联 / 大钥 copy_from_user）
     ▼
[内核] 存好身份与 peer ── 配置完成，用户态可退场
     │
     │ （用户发流量 / keepalive 作为触发）
     ▼
[内核 I] ── Initiation UDP ──► [内核 R]   交换：E_i^pub, ct1, Enc(Hash(Si)), Enc(t)
[内核 I] ◄── Response UDP ─── [内核 R]   交换：ct2, ct3, Enc(∅)
[内核 I] ── Key Confirm UDP ─► [内核 R]   交换：Enc(∅)=conf
     │
     ▼
[内核双方] 持有 TK_i/TK_r，开始数据面
     │
     │ （可选）Netlink GET：LastHandshakeTime 等
     ▼
[用户态] 仅看状态，不碰会话密钥材料
```

## 4. 收口表


| 步骤         | 用户态            | 内核态                | 交换信息                           | 拷贝（用户态↔内核）                 |
| ---------- | -------------- | ------------------ | ------------------------------ | -------------------------- |
| 配钥         | genkey、写 conf  | —                  | —                              | 无                          |
| 下发配置       | `pqst set`     | 收 Netlink，存钥与 peer | 配置属性                           | **本机钥、对端公钥、PSK、endpoint…** |
| 触发         | 发包 / keepalive | 决定发起握手             | —                              | 一般无额外钥拷贝                   |
| Initiation | —              | 组包/收包/KEM          | `E_i^{pub}, ct1, static, time` | **无**                      |
| Response   | —              | 组包/收包/KEM          | `ct2, ct3, empty`              | **无**                      |
| Confirm    | —              | 确认并派生 TK           | `conf`                         | **无**                      |
| 之后         | 可选 `show`      | 数据面                | 隧道数据包                          | 仅状态；无会话钥                   |


---



## 5. 总结

> 用户态负责把「谁是谁、连哪、允许哪些网段」通过 Netlink 拷进内核；内核单独完成「认服务器 → 认本轮会话 → 认客户端」三次 KEM 与 UDP 报文交换。整场握手期间，用户态与内核之间不再为握手字段做拷贝。
