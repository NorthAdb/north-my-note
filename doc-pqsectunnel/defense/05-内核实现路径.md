# 05 — 内核实现路径

> 状态：正文 | 权威源：[KE.md](../../KE.md) L3 + `receive.c` / `send.c` / `noise.c`  
> 主套件：`mlkem512-mlkem768-sm4gcm-sm3` | 预计篇幅：5 min 讲稿（30 min 版）

## 1. 核心原则

KE.md L3：

> mac1/mac2 不参与 C/H 链；**接收侧 receive.c 入口先验 mac**，再进 noise.c 做 KEM/AEAD。

KE 表格中的 mac 步骤是**发送侧组包/transcript 顺序**；noise.c 内只做 KEM、AEAD 与 C/H 更新，不算 MAC。

---

## 2. 内核模块文件分工

| 文件 | 职责 |
|------|------|
| `noise.c` / `noise.h` | 握手状态机、KEM/AEAD、C/H 链、会话密钥派生 |
| `messages.h` | 报文 struct、尺寸宏、type 枚举 |
| `send.c` | 握手发送调度、MAC 添加、应用层分片发送 |
| `receive.c` | UDP 收包、分片重组、MAC 验证、握手 dispatch、数据面解密 |
| `cookie.c` | MAC1/MAC2 计算与 Cookie Reply |
| `pqc_fragment.c` | 应用层分片/重组（主套件 Initiation/Response） |
| `socket.c` | 原始 UDP 发送（`send4`/`send6`） |
| `device.c` / `peer.c` | 设备与 peer 生命周期 |
| `allowedips.c` | 数据面路由 Trie |
| `timers.c` | keepalive、rekey、握手重试 |

密码运算**全部在内核** `pqsectunnel.ko`；用户态 **pqst** 仅通过 Netlink 配置密钥与 peer（CONTEXT **pqsectunnel.ko** / **pqst**）。

## 3. 握手状态机

定义：`noise.h` `enum noise_handshake_state`

| 状态 | 含义 | 典型触发 |
|------|------|---------|
| `HANDSHAKE_ZEROED` | 初始/已清零 | 握手开始或 `begin_session` 后 |
| `HANDSHAKE_CREATED_INITIATION` | Initiator 已发 Initiation | `create_initiation` 成功 |
| `HANDSHAKE_CONSUMED_INITIATION` | Responder 已消费 Initiation | `consume_initiation` 成功 |
| `HANDSHAKE_CREATED_RESPONSE` | Responder 已发 Response | `create_response` 成功 |
| `HANDSHAKE_CONSUMED_RESPONSE` | Initiator 已消费 Response | `consume_response` 成功 |

状态转移与三报文对应关系见 [03](03-握手协议逐步详解.md) §12.3。

## 4. 接收路径（握手）

### 4.1 总入口

```
UDP 到达
  → wg_packet_receive()                    [receive.c L602]
  → prepare_skb_header()                   校验 UDP 载荷
  → pqc_fragment_recv()                    若 magic=0x5051，重组 [L607–627]
  → 入队 handshake_queue                   type 1–4 [L631–657]
  → wg_receive_handshake_packet()          工作队列处理 [L106]
```

### 4.2 握手包处理（MAC 先于 noise）

`wg_receive_handshake_packet()`（L106–219）顺序：

| 步 | 操作 | 源码行 |
|----|------|--------|
| 1 | 若为 Cookie Reply (type=4)，单独处理 | L118–123 |
| 2 | 判断高负载 `under_load` | L126–134 |
| 3 | **`wg_cookie_validate_packet()` 验 MAC** | **L135–136** |
| 4 | mac₂ 无效且高负载 → 发 Cookie，**不进 noise** | L140–156 等 |
| 5 | `wg_noise_handshake_consume_*()` | L158 / L180 / L202 |
| 6 | 成功后触发对端发送下一报文 | L168 / L190 / L212 |

### 4.3 三报文 consume 与后续动作

| type | consume 函数 | 成功后 |
|------|-------------|--------|
| 1 Initiation | `wg_noise_handshake_consume_initiation` | `wg_packet_send_handshake_response` |
| 2 Response | `wg_noise_handshake_consume_response` | `wg_packet_send_handshake_key_confirm` |
| 3 Key Confirm | `wg_noise_handshake_consume_key_confirm` | `wg_noise_handshake_begin_session` + timers |

Key Confirmation 接收侧（L212–217）：`begin_session` → `pqc_fragment_session_established` → `wg_timers_session_derived` → keepalive。

## 5. 发送路径（握手）

### 5.1 三报文发送链

| 报文 | 组包 (noise) | + MAC (cookie) | 发送 |
|------|-------------|----------------|------|
| Initiation | `wg_noise_handshake_create_initiation` | `wg_cookie_add_mac_to_packet` | `pqc_fragment_send` 或 `wg_socket_send_buffer_to_peer` |
| Response | `wg_noise_handshake_create_response` | 同上 | 同上 |
| Key Confirm | `wg_noise_handshake_create_key_confirm` | 同上 | **单包** `wg_socket_send_buffer_to_peer`（60 B） |

源码：`send.c` L50–74（Initiation）、L120–140（Response）、L144–169（Key Confirm）。

### 5.2 发送顺序要点

1. **noise.c 组包** — 填充 struct，更新 C/H，不含 MAC 字段
2. **`wg_cookie_add_mac_to_packet`** — MAC 覆盖 type…payload（完整握手缓冲）
3. **分片或直发** — 主套件 Initiation/Response 走 `pqc_fragment_send`（`#if PQST_APP_LAYER_FRAGMENTATION`）
4. **Key Confirm 特殊** — Initiator 在 `create_key_confirm` 后、发送前调用 `begin_session`（L155–156）

MAC 必须在分片**之前**计算，保证每个分片重组后的缓冲与 MAC 验证一致（见 [06](06-分片与MTU.md)）。

## 6. 数据面路径（MESSAGE_DATA）

握手完成后 type=5：

**接收**（`receive.c`）：
```
wg_packet_receive → type=5
  → wg_packet_consume_data
  → 查找 keypair（index + replay counter）
  → 入队 decrypt_queue → AEAD 解密（SM4-GCM）
  → allowedips 匹配 → netif_rx 注入
```

**发送**（`send.c` / `queueing.c`）：
```
明文 IP 包 → allowedips 选 peer
  → AEAD 加密（TK + counter）
  → wg_socket_send_skb_to_peer
```

会话密钥 **TK_i / TK_r** 由 `derive_keys()` 从 C₁₀ 派生；Initiator/Responder 的 sending/receiving 方向在 `begin_session` 中按状态区分（[03](03-握手协议逐步详解.md) §9.1）。

---

## 7. 密钥层次（握手 vs 数据面）

| 密钥 | 派生阶段 | 用途 | noise.c |
|------|---------|------|---------|
| K₁ | Initiation（ct₁ 后） | AEAD 加密 static | `mix_kem_ss` → `message_encrypt(static)` |
| K₂ | Initiation（mix_psk 后） | AEAD 加密 time | `mix_psk` → `message_encrypt(time)` |
| K₃ | Response（ct₂/ct₃ 后） | AEAD 加密 empty | `mix_psk` → `message_encrypt(empty)` |
| K₄ | Response/Confirm（empty 后） | AEAD 加密 conf | `kdf` → `message_encrypt(conf)` |
| **TK_i / TK_r** | begin_session | MESSAGE_DATA | `derive_keys()` |

CONTEXT：**K1–K4** vs **会话密钥（TK_i / TK_r）**。

## 8. noise.c 入口速查

完整逐步映射见 [03](03-握手协议逐步详解.md) §12。

| 函数 | 角色 |
|------|------|
| `wg_noise_handshake_create_initiation` | Initiator 组 Initiation |
| `wg_noise_handshake_consume_initiation` | Responder 解 Initiation |
| `wg_noise_handshake_create_response` | Responder 组 Response |
| `wg_noise_handshake_consume_response` | Initiator 解 Response |
| `wg_noise_handshake_create_key_confirm` | Initiator 组 Key Confirm |
| `wg_noise_handshake_consume_key_confirm` | Responder 解 Key Confirm |
| `wg_noise_handshake_begin_session` | 派生 TK、安装 keypair、清零 handshake |

Transcript 原语：`handshake_init`、`message_ephemeral`、`mix_hash`、`mix_kem_ss`、`mix_psk`、`mix_key_chain_only`、`message_encrypt/decrypt`、`kdf`、`noise_hash`。

## 9. 主套件端到端调用链

```
Initiator (pqst0)                         Responder (server0.pqst0)
─────────────────                         ───────────────────────
send: create_initiation
      add_mac
      pqc_fragment_send (2068→2片)  ────► recv: fragment reasm (2068)
                                           validate_mac
                                           consume_initiation
                                      ◄─── send: create_response
                                           add_mac
                                           pqc_fragment_send (1916→2片)
recv: fragment reasm (1916)
      validate_mac
      consume_response
send: create_key_confirm
      begin_session (TK)
      add_mac
      send (60B, 无分片)          ────► recv: validate_mac
                                           consume_key_confirm
                                           begin_session (TK)
      ◄════ MESSAGE_DATA ════════►
```

主套件 dmesg 实证：`experiments/mlkem512-mlkem768-sm4gcm-sm3/dmesg.log` L87–123。
