# 03 — 握手协议逐步详解

> 状态：正文 | 权威源：[KE.md](../../KE.md)、[CONTEXT.md](../../CONTEXT.md)、`noise.c`  
> 主套件：`mlkem512-mlkem768-sm4gcm-sm3`（Static ML-KEM-512 + Ephemeral ML-KEM-768 + SM4-GCM + SM3）

## 1. 概述

PQ-SecTunnel 采用 **Noise_IKpsk1kem** 变体：相对 WireGuard 的 Noise_IKpsk2，将 X25519 ECDH 替换为 **双 KEM 握手**（Static KEM + Ephemeral KEM），并增加第三轮 **Key Confirmation**。

一次完整握手为 **1.5-RTT**：

| 轮次 | 报文 | 方向 | 类型 |
|------|------|------|------|
| 第 1 报文 | Handshake Initiation | Initiator → Responder | type=1 |
| 第 2 报文 | Handshake Response | Responder → Initiator | type=2 |
| 第 3 报文 | Key Confirmation | Initiator → Responder | type=3 |

主套件握手 Noise 标识（编译时写入模块）：

```
Noise_IKpsk1kem_SM4GCM_SM3_EphemMLKEM768
```

来源：`wgtest/wgtest-mlkem512-mlkem768-sm4gcm-sm3/vars` → `NOISE_HANDSHAKE_NAME`。

---

## 2. 主套件报文尺寸与字段作用

结构定义见 `wireguard-linux-compat/src/messages.h`。主套件：`mlkem512-mlkem768-sm4gcm-sm3`（Static **768 B** 密文 / Ephemeral **1184 B** 公钥与 **1088 B** 密文；AEAD 标签恒 **16 B**）。

合计：Initiation **2068 B**、Response **1916 B**、Key Confirmation **60 B**。

### 2.1 Handshake Initiation — 2068 B（Initiator → Responder，type=1）

| 字段 | 字节 | 作用（一句话） |
|------|------|----------------|
| **header** | 4 | `type=1` + reserved；区分报文种类 |
| **sid_i** | 4 | Initiator 会话索引；Response/Confirm 用它回指本轮握手 |
| **E_i^pub** | 1184 | 本轮临时 ML-KEM-768 公钥（明文）；供 Responder 做 ct₂，并混入 transcript |
| **ct₁** | 768 | Static ML-KEM-512 对 **S_r^pub** 的封装（≈ 原版 `es`）：认证 Responder，派生 K₁ |
| **static** | 48 | SM4-GCM 加密的 **Hash(S_i^pub)**（32+16 tag）：延迟披露 Initiator 身份，供 peer 查找 |
| **time** | 28 | SM4-GCM 加密的 TAI64N 时间戳（12+16 tag）：防重放 / 限速发起 |
| **mac_1** | 16 | DoS 初筛：绑 `S_r^pub`；**不进** C/H 链 |
| **mac_2** | 16 | 高负载时基于 Cookie；未用则为 0；同样不进 C/H |

字段字节：`4+4+1184+768+48+28+16+16 = 2068`。

### 2.2 Handshake Response — 1916 B（Responder → Initiator，type=2）

| 字段 | 字节 | 作用（一句话） |
|------|------|----------------|
| **header** | 4 | `type=2` + reserved |
| **sid_r** | 4 | Responder 会话索引；后续 Confirm/数据关联本端状态 |
| **sid_i** | 4 | 回显 Initiation 的 sid；把本包绑到那一轮发起 |
| **ct₂** | 1088 | Ephemeral ML-KEM-768 对 **E_i^pub** 的封装（≈ 原版 `ee`）：会话绑定 + 前向保密 |
| **ct₃** | 768 | Static ML-KEM-512 对 **S_i^pub** 的封装（≈ 原版 `se`）：认证 Initiator |
| **empty** | 16 | SM4-GCM 空载荷（仅 16 B tag）：证明 Responder 已持 K₃，并推进到 C₁₀/K₄ |
| **mac_1** | 16 | DoS MAC；不进 C/H |
| **mac_2** | 16 | Cookie MAC 或 0 |

字段字节：`4+4+4+1088+768+16+16+16 = 1916`。  
此时双方 transcript 材料已齐，**可各自推导同一 C₁₀**（Key Confirm 不再贡献新 KEM 秘密）。

### 2.3 Key Confirmation — 60 B（Initiator → Responder，type=3）

| 字段 | 字节 | 作用（一句话） |
|------|------|----------------|
| **header** | 4 | `type=3` + reserved |
| **sid_i** | 4 | Initiator 索引 |
| **sid_r** | 4 | Responder 索引；指明确认哪一轮 |
| **conf** | 16 | SM4-GCM 用 **K₄** 加密的空载荷：**显式密钥确认**（证明 Initiator 也算出了正确密钥） |
| **mac_1** | 16 | DoS MAC |
| **mac_2** | 16 | Cookie MAC 或 0 |

字段字节：`4+4+4+16+16+16 = 60`。  
Responder 解 conf 成功后再 `begin_session`；Initiator 侧多在发 Confirm 前已装会话（见 §8–9）。

### 2.4 分片与对照

| 报文 | 是否 >1400 B | 应用层分片 |
|------|--------------|------------|
| Initiation 2068 | 是 | **2 片**（`APP_LAYER_FRAGMENTATION=auto`） |
| Response 1916 | 是 | **2 片** |
| Key Confirmation 60 | 否 | 单 UDP 包 |

详见 [06](06-分片与MTU.md)；尺寸推导与 KEM 参数见 [04](04-密码原语与套件.md)。

## 3. Transcript（C/H 链）初始化

模块加载时 `wg_noise_init()` 用 SM3 预计算协议标签（对应 KE.md L6–7）：

| KE 步骤 | 含义 | noise.c 实现 |
|---------|------|-------------|
| C₀ = Hash(label₁) | 初始 Chaining Key | `SM3(handshake_name)` → `handshake_init_chaining_key` |
| H₀ = Hash(C₀ ‖ label₂) | 初始 Hash | `SM3(C₀ ‖ identifier_name)` → `handshake_init_hash` |

主套件 `handshake_name` = `Noise_IKpsk1kem_SM4GCM_SM3_EphemMLKEM768`；`identifier_name` = `PQWireGuard v2 zx2c4 Jason@zx2c4.com`（vars 中 `NOISE_IDENTIFIER_NAME`）。

每次握手开始时 `handshake_init()` 将 Responder 静态公钥混入 Transcript（KE.md L8）：

| KE 步骤 | 操作 | noise.c |
|---------|------|---------|
| H₁ = Hash(H₀ ‖ S_r^pub) | 混入 Responder 静态公钥 | `mix_hash(hash, remote_static, …)` |
| C₁ = KDF(C₀, S_r^pub) | 更新 Chaining Key | `mix_key_chain_only(chaining_key, remote_static, …)` |

## 4. Handshake Initiation（Initiator 发送）

**入口**：`send.c` → `wg_noise_handshake_create_initiation()` → `wg_cookie_add_mac_to_packet()` →（可选）`pqc_fragment_send()`。

### 4.1 逐步对照

| KE | 操作 | noise.c 函数 / 代码位置 |
|----|------|-------------------------|
| L5 | sid_i ← sender_index | `wg_index_hashtable_insert()`（组包末尾，L1105–1107） |
| L6–8 | C/H 初始化，绑定 S_r^pub | `handshake_init()` |
| L9–10 | 生成 E_i，混入 C/H | `pqst_ephemeral_keygen()` → `kem768_keygen()`；`message_ephemeral()` |
| L11 | ct₁ = Enc(S_r^pub) | `kem_encapsulate()`（Static ML-KEM-512） |
| L12–13 | H₃ = Hash(H₂‖ct₁)；KDF(C₂, Hash(ct₁)) | `mix_hash()`；`noise_hash(ss, ct₁)` **覆写 shk**；`mix_kem_ss()` → 派生 K₁ |
| L14 | static = AEAD.Enc(K₁, Hash(S_i^pub)) | `message_encrypt(encrypted_static, initiator_public_hash, …)` |
| L15 | H₄ = Hash(H₃‖static) | `message_encrypt()` 内部 `mix_hash()` |
| L16–17 | C₄ = KDF(C₃, S_i^pub)；(C₅, K₂) = KDF(C₄, H₄‖psk) | `mix_key_chain_only(chaining_key, initiator_public, …)`；`mix_psk()` |
| L18 | time = AEAD.Enc(K₂, t_now) | `tai64n_now()`；`message_encrypt(encrypted_timestamp, …)` |
| L19–24 | mac₁ / mac₂ | **不在 noise.c**；`send.c` 中 `wg_cookie_add_mac_to_packet()` |

### 4.2 主套件 KEM 调用

| 操作 | 算法 | API |
|------|------|-----|
| 生成 E_i | ML-KEM-768 | `pqst_ephemeral_keygen` → `kem768_keygen` |
| Enc(S_r^pub) → ct₁ | ML-KEM-512 | `kem_encapsulate()` |

Ephemeral 宏在 `noise.c` L32–35：当 `EPHEMERAL_KEM_MLKEM768` 定义时映射到 `kem768_*`。

### 4.3 明文载荷

- **static** 字段加密的是 `Hash(S_i^pub)`（`initiator_public_hash`），由 `digest_pk()` 预计算；主套件使用 SM3-HMAC（`noise.c` L1121–1123）。
- **time** 为 TAI64N 格式时间戳（`tai64n_now()`），Responder 用于重放检测。

### 4.4 PSK

若配置中 PSK 全零，`wg_noise_handshake_init()` 调用 `wg_noise_maybe_set_default_psk()`（KE.md L2）：

```
psk = Hash(S_i^pub XOR S_r^pub)   // 按字节异或 Responder 静态公钥后 SM3 哈希
```

## 5. Handshake Initiation（Responder 接收）

**入口**：`receive.c` 先 `wg_cookie_validate_packet()` 验 MAC，再 `wg_noise_handshake_consume_initiation()`。

| KE | 操作 | noise.c |
|----|------|---------|
| L28–31 | 重建 C/H 至 E_i | `handshake_init()`；`message_ephemeral()` |
| L32–34 | Dec(S_r^priv, ct₁)；静态 KEM 混链 | `kem_decapsulate()`；`noise_hash(ss, ct₁)`；`mix_kem_ss()` |
| L35–38 | 解密 static → Hash(S_i^pub)；mix S_i；mix_psk | `message_decrypt(encrypted_static, …)`；`mix_key_chain_only(S_i^pub)`；`mix_psk()` |
| L39 | 解密 time | `message_decrypt(encrypted_timestamp, …)` |
| L41–48 | 验 mac₁/mac₂ | **receive.c L135**（noise 入口之前） |

解密 static 后，Responder 用 `Hash(S_i^pub)` 在 peer 表查找 Initiator（`wg_pubkey_hashtable_lookup()`），并将 transcript 状态写入 peer 的 `noise_handshake`（状态 → `HANDSHAKE_CONSUMED_INITIATION`）。

---

## 6. Handshake Response（Responder 发送）

**入口**：`receive.c` 消费 Initiation 成功后 → `wg_packet_send_handshake_response()` → `wg_noise_handshake_create_response()`。

| KE | 操作 | noise.c |
|----|------|---------|
| L49–51 | ct₂ = Enc(E_i^pub)；混入 ct₂、shk₂ | `pqst_ephemeral_encapsulate()` → `kem768_encapsulate()`；`mix_hash` + `mix_key_chain_only(ct₂)` + `mix_kem_ss(shk₂)` |
| L52–54 | ct₃ = Enc(S_i^pub)；静态 KEM 混链 | `kem_encapsulate()`；`noise_hash(ss, ct₃)`；`mix_kem_ss()` |
| L55–58 | mix_psk；AEAD 加密 empty | `mix_psk()`；`message_encrypt(encrypted_nothing, NULL, 0, …)` |
| L59–60 | (C₁₀, K₄) = KDF(C₉, empty) | `kdf(chaining_key, key, NULL, NULL, …)` |
| L61–64 | mac₁ / mac₂ | `send.c` → `wg_cookie_add_mac_to_packet()` |

ct₂ 使用 **Ephemeral ML-KEM-768**（1088 B）；ct₃ 使用 **Static ML-KEM-512**（768 B）。

组包完成后状态 → `HANDSHAKE_CREATED_RESPONSE`；`sid_r` 由 `wg_index_hashtable_insert()` 分配。

---

## 7. Handshake Response（Initiator 接收）

**入口**：`receive.c` 验 MAC → `wg_noise_handshake_consume_response()` → 触发 `wg_packet_send_handshake_key_confirm()`。

| KE | 操作 | noise.c |
|----|------|---------|
| L76–78 | Dec(E_i^priv, ct₂) | `pqst_ephemeral_decapsulate()` → `kem768_decapsulate()`；`mix_kem_ss(shk₂)` |
| L79–81 | Dec(S_i^priv, ct₃)；静态混链 | `kem_decapsulate()`；`noise_hash(ss, ct₃)`；`mix_kem_ss()` |
| L82–85 | mix_psk；解密 empty | `mix_psk()`；`message_decrypt(encrypted_nothing, …)` |
| L86 | (C₁₀, K₄) = KDF(C₉, empty) | `kdf(chaining_key, key, NULL, NULL, …)` |
| L68–75 | 验 mac₁/mac₂ | **receive.c**（noise 入口之前） |

成功后状态 → `HANDSHAKE_CONSUMED_RESPONSE`；Initiator 侧已具备派生 **TK_i / TK_r** 所需的 `chaining_key`（C₁₀）。

---

## 8. Key Confirmation（Initiator 发送）

**入口**：`wg_noise_handshake_create_key_confirm()`。

| KE | 操作 | noise.c |
|----|------|---------|
| L86 | conf = AEAD.Enc(K₄, ∅) | `message_encrypt(encrypted_conf, NULL, 0, key, hash)` |
| L87–92 | mac₁ / mac₂ | `send.c` → `wg_cookie_add_mac_to_packet()` |

Initiator 在**发送前**即调用 `wg_noise_handshake_begin_session()`（`send.c` L155–156），从 C₁₀ 派生会话密钥并安装 keypair；Key Confirmation 报文本身仅 60 B，不经应用层分片。

---

## 9. Key Confirmation（Responder 接收）与会话建立

**入口**：`receive.c` → `wg_noise_handshake_consume_key_confirm()` → `wg_noise_handshake_begin_session()`。

| KE | 操作 | noise.c |
|----|------|---------|
| L97–104 | 验 mac₁/mac₂ | **receive.c** |
| L105 | 解密 conf | `message_decrypt(encrypted_conf, …)` |
| L106 | (TK_i, TK_r) = KDF(C₁₀, ∅) | `wg_noise_handshake_begin_session()` → `derive_keys()` → `kdf(TK_send, TK_recv, ∅, chaining_key)` |

### 9.1 会话密钥方向

`derive_keys()` 对 C₁₀ 做 `KDF(∅)`，得到两个 128-bit SM4-GCM 密钥（主套件 `NOISE_SYMMETRIC_KEY_LEN=16`）：

| 角色 | begin_session 时状态 | sending | receiving |
|------|---------------------|---------|-----------|
| Initiator | `HANDSHAKE_CONSUMED_RESPONSE` | TK_i | TK_r |
| Responder | `HANDSHAKE_CREATED_RESPONSE` | TK_r | TK_i |

安装 keypair 后 `handshake_zero()` 清零握手状态，进入 **MESSAGE_DATA** 数据面。

## 10. MAC1/MAC2 与 C/H 链的关系

KE.md L3 明确：**mac1/mac2 不参与 C/H 链**。

| 环节 | 位置 | 行为 |
|------|------|------|
| MAC 计算 | `cookie.c` / `send.c` | 覆盖完整握手 struct（type…payload，不含 mac 字段本身） |
| MAC 验证 | `receive.c` L135 | `wg_cookie_validate_packet()`，**先于** noise 握手 consume |
| C/H 更新 | `noise.c` | `mix_hash`、`mix_kem_ss`、`mix_psk`、`message_encrypt/decrypt` |

MAC1 密钥派生自 `BLAKE2s("mac1----" ‖ S_r^pub)`（与 **PSK** 无关，见 CONTEXT.md **MAC1** / **PSK**）。

高负载下若 mac2 无效，Responder 返回 Cookie Reply（type=4），Initiator 重发带有效 mac2 的报文——见时序图可选分支。

## 11. 静态 KEM 混链特殊规则

KE.md L109 与 `noise.c` 一致：**ct₁、ct₃** 在 encaps/decaps 得到 raw shk 后，调用

```c
noise_hash(shared_secret, ciphertext, …);  // shk ← Hash(ct)
mix_kem_ss(chaining_key, key, shared_secret);
```

**ct₂** 路径直接使用 raw `shk₂` 进入 `mix_kem_ss()`，**不**做 `noise_hash` 覆写。此为当前实现的刻意行为，未来对齐 spec 属 BREAKING 变更 → [09](09-已知差异与答辩备查.md)。

`noise_hash()` 在 Initiation/Response 中共 **4 处**（ct₁/ct₃ 各 encaps 与 decaps 侧各 1 处）：

| 位置 | 函数 | 行号（约） |
|------|------|-----------|
| Initiator 发 Initiation | `wg_noise_handshake_create_initiation` | L1065–1066 |
| Responder 收 Initiation | `wg_noise_handshake_consume_initiation` | L1203–1204 |
| Responder 发 Response | `wg_noise_handshake_create_response` | L1365–1366 |
| Initiator 收 Response | `wg_noise_handshake_consume_response` | L1489–1490 |

## 12. noise.c 函数映射总表

### 12.1 模块初始化与握手生命周期

| 函数 | 作用 | 对应 KE / 说明 |
|------|------|---------------|
| `wg_noise_init()` | 预计算 C₀、H₀（label 哈希） | KE L6–7 |
| `wg_noise_handshake_init()` | 初始化 peer handshake；默认 PSK | KE L2；调用 `wg_noise_maybe_set_default_psk()` |
| `wg_noise_maybe_set_default_psk()` | 未配置 PSK 时异或静态公钥哈希 | KE L2 |
| `wg_noise_handshake_clear()` | 清除握手状态 | — |
| `wg_noise_handshake_begin_session()` | KDF(C₁₀,∅) → TK；安装 keypair | KE L94/L106 |
| `derive_keys()` | 派生 TK_i、TK_r（SM4-GCM key） | KE L94/L106 |

### 12.2 Transcript 原语

| 函数 | 作用 | KE 对应 |
|------|------|--------|
| `handshake_init()` | H₁、C₁：混入 S_r^pub | L8 |
| `message_ephemeral()` | H₂、C₂：混入 E_i^pub | L10 |
| `mix_hash()` | H 链更新 | 各 H 步骤 |
| `mix_key_chain_only()` | 仅更新 C（KDF 单输出） | C₄ = KDF(C₃, S_i) 等 |
| `mix_kem_ss()` | KDF(C, shk) → 新 C + 对称密钥 K | K₁–K₄ 派生 |
| `mix_psk()` | mix_psk：KDF(C, H‖psk) | L16–17 等 |
| `kdf()` | Noise 标准 KDF（HMAC-SM3 主套件） | 全部 KDF |
| `noise_hash()` | Hash；静态 KEM shk 覆写 | L109 脚注 |
| `message_encrypt()` | SM4-GCM 加密 + mix_hash | static/time/empty/conf |
| `message_decrypt()` | SM4-GCM 解密 + mix_hash | 对端解密 |
| `digest_pk()` / `digest_pk_responder()` | Hash(S_i^pub) / Hash(S_r^pub) | static 字段明文 |

### 12.3 三报文入口（noise 层）

| 方向 | 组包 | 解包 | send/receive 调度 |
|------|------|------|------------------|
| Initiation | `wg_noise_handshake_create_initiation` | `wg_noise_handshake_consume_initiation` | `send.c` L63 / `receive.c` L158 |
| Response | `wg_noise_handshake_create_response` | `wg_noise_handshake_consume_response` | `send.c` L129 / `receive.c` L180 |
| Key Confirmation | `wg_noise_handshake_create_key_confirm` | `wg_noise_handshake_consume_key_confirm` | `send.c` L153 / `receive.c` L202 |

### 12.4 KEM API（主套件）

| 角色 | 操作 | Static（512） | Ephemeral（768） |
|------|------|---------------|------------------|
| Initiator 发 | keygen / encaps | `kem_encapsulate` → ct₁ | `kem768_keygen` → E_i |
| Responder 收 | decaps | `kem_decapsulate` ← ct₁ | （读 E_i^pub） |
| Responder 发 | encaps | `kem_encapsulate` → ct₃ | `kem768_encapsulate` → ct₂ |
| Initiator 收 | decaps | `kem_decapsulate` ← ct₃ | `kem768_decapsulate` ← ct₂ |

宏别名：`pqst_ephemeral_*` → `kem768_*`（`EPHEMERAL_KEM_MLKEM768`）。

### 12.5 MAC 与发送（noise 层之外）

| 函数 | 文件 | 作用 |
|------|------|------|
| `wg_cookie_add_mac_to_packet()` | `send.c` / `cookie.c` | 组包后计算 mac₁/mac₂ |
| `wg_cookie_validate_packet()` | `receive.c` / `cookie.c` | 收包入口验 MAC |
| `pqc_fragment_send()` | `send.c` | 主套件 Initiation/Response 应用层分片 |

## 13. 端到端调用链（主套件）

```
Initiator                                    Responder
    |                                            |
    |-- create_initiation (noise.c) ------------>|
    |   + add_mac (send.c)                       |
    |   + pqc_fragment_send (2068B)              |
    |                                            | validate_mac (receive.c)
    |                                            | consume_initiation (noise.c)
    |                                            | create_response (noise.c)
    |                                            | add_mac + pqc_fragment_send
    |<----------- Response (1916B) -------------|
    | validate_mac                               |
    | consume_response (noise.c)                 |
    | create_key_confirm (noise.c)               |
    | begin_session → TK_i/TK_r                  |
    | add_mac                                    |
    |-- Key Confirmation (60B) ---------------->|
    |                                            | validate_mac
    |                                            | consume_key_confirm
    |                                            | begin_session → TK_i/TK_r
    |<=========== MESSAGE_DATA =================>|
```
