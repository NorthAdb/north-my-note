# 02 — 协议总览

> 状态：正文 | 权威源：时序图 + [KE.md](../../KE.md) + [CONTEXT.md](../../CONTEXT.md)  
> 主套件：`mlkem512-mlkem768-sm4gcm-sm3` | 预计篇幅：3 min 讲稿

## 1. 协议身份

PQ-SecTunnel 使用 **Noise_IKpsk1kem** 握手模式（CONTEXT **Noise_IKpsk1kem**）：在 Noise IKpsk 框架上引入 KEM 封装/解封装，并增加 **Key Confirmation** 第三轮，形成相对 WireGuard **Noise_IKpsk2** 的三次握手变体。

核心密码结构为 **双 KEM 握手**（CONTEXT **双 KEM 握手**）：


| KEM 角色            | 主套件算法      | 握手中的载体      |
| ----------------- | ---------- | ----------- |
| **Static KEM**    | ML-KEM-512 | ct₁、ct₃     |
| **Ephemeral KEM** | ML-KEM-768 | E_i^pub、ct₂ |


对称层使用 **SM4-GCM**（AEAD）与 **SM3**（Hash/KDF）；详见 [04](04-密码原语与套件.md)。

逐步密码运算见 [03](03-握手协议逐步详解.md)；本节只讲**阶段划分、角色分工与报文轮廓**。

## 3. 三阶段握手


| 阶段     | 报文                       | 方向                    | type | 主套件尺寸  | KE.md   |
| ------ | ------------------------ | --------------------- | ---- | ------ | ------- |
| 第 1 报文 | **Handshake Initiation** | Initiator → Responder | 1    | 2068 B | L4–25   |
| 第 2 报文 | **Handshake Response**   | Responder → Initiator | 2    | 1916 B | L27–65  |
| 第 3 报文 | **Key Confirmation**     | Initiator → Responder | 3    | 60 B   | L67–94  |
| 完成     | 派生 **TK_i / TK_r**       | 双方                    | —    | —      | L94–106 |


**1.5-RTT** 含义：Initiator 发出 Initiation 后，经过 Response 往返，再发送 Key Confirmation；第三条报文与第二条 Response 在时间上部分重叠（Initiator 收到 Response 后立即发 Confirm），整体少于 2 个完整 RTT。

### 3.1 第 1 报文 — Handshake Initiation

Initiator 发送，主要携带：


| 字段           | 含义                                        |
| ------------ | ----------------------------------------- |
| sid_i        | Initiator 会话索引                            |
| E_i^pub      | 临时 KEM 公钥（ML-KEM-768）                     |
| ct₁          | 对 Responder **Static KEM 公钥** S_r^pub 的封装 |
| static       | AEAD 加密的 Hash(S_i^pub)                    |
| time         | AEAD 加密的时间戳                               |
| mac_1, mac_2 | DoS 防护 MAC                                |


Responder 收到后：验 MAC → 解封 ct₁ → 识别 Initiator → 准备 Response。  

> 时序图中「ctl」为口语缩写，规范术语为 **ct₁**（CONTEXT **ct₁**）。



### 3.2 第 2 报文 — Handshake Response

Responder 发送，主要携带：


| 字段           | 含义                                          |
| ------------ | ------------------------------------------- |
| sid_r, sid_i | Responder 索引；关联 Initiation                  |
| ct₂          | 对 Initiator 临时公钥 E_i^pub 的封装（Ephemeral KEM） |
| ct₃          | 对 Initiator **Static KEM 公钥** S_i^pub 的封装   |
| empty        | AEAD 加密空载荷（证明持有 K₃）                         |
| mac_1, mac_2 | DoS 防护 MAC                                  |


Initiator 收到后：验 MAC → 解封 ct₂、ct₃ → 验证 empty → 派生至 C₁₀/K₄，准备 Key Confirmation。

Response 完成后双方 transcript 上已可推导会话密钥材料，但 **Key Confirmation** 提供显式确认（CONTEXT **Key Confirmation**）。

### 3.3 第 3 报文 — Key Confirmation

Initiator 发送，主要携带：


| 字段           | 含义           |
| ------------ | ------------ |
| sid_i, sid_r | 双向索引         |
| conf         | AEAD 加密空确认载荷 |
| mac_1, mac_2 | DoS 防护 MAC   |


Responder 验 conf 后，双方执行 **(TK_i, TK_r) = KDF(C₁₀, ∅)**，进入数据面。

## 4. 角色分工


| 角色            | 职责                                                | CONTEXT       |
| ------------- | ------------------------------------------------- | ------------- |
| **Initiator** | 生成 E_i；发 Initiation；收 Response；发 Key Confirmation | **Initiator** |
| **Responder** | 收 Initiation；发 Response；收 Key Confirmation        | **Responder** |


Initiator / Responder 是**协议角色**，与部署上的「客户端/服务端」无固定对应——任意 peer 均可发起握手。

### 4.1 会话密钥建立时机


| 方         | begin_session 触发点                  | 源码          |
| --------- | ---------------------------------- | ----------- |
| Initiator | 组包 Key Confirmation **之后、发送之前**    | `send.c`    |
| Responder | 收 Key Confirmation **并解密 conf 之后** | `receive.c` |


双方均从 C₁₀ 派生 **TK_i / TK_r**（CONTEXT **会话密钥**），用于后续 **MESSAGE_DATA** AEAD 保护。

---



## 5. 离线准备

握手开始前（时序图顶部「离线准备」），双方需完成：

1. **Peer 记录**：配置对端 Static KEM 公钥（或 PKI 证书提取公钥）
2. **本地 Static KEM 密钥对**：S_i 或 S_r 的长期身份密钥
3. **可选 PSK**：未配置时默认值为双方 Static KEM 公钥异或后的 Hash（KE.md L2）

以上通过用户态 **pqst** / **pqst-quick** 经 Netlink 下发内核；运行时握手密码运算在 **pqsectunnel.ko** 内完成（CONTEXT **pqsectunnel.ko** / **pqst**）。

## 6. 可选 Cookie 分支（高负载防护）

时序图红色虚线所示，仅在 Responder **高负载**且 Initiator **mac₂ 无效**时触发：

```
Initiator ──Initiation(mac₂=0)──► Responder
                                      │ 验 mac₁ OK，mac₂ 无效
Initiator ◄── Cookie Reply(type=4) ──┘
Initiator ──Initiation(mac₂=Cookie_MAC)──► Responder
                                      │ 继续正常握手
```


| 项            | 说明                                                        |
| ------------ | --------------------------------------------------------- |
| Cookie Reply | type=4（`MESSAGE_HANDSHAKE_COOKIE`），`cookie.c` 组包          |
| 入口           | `receive.c`：高负载时 `wg_cookie_validate_packet()` 返回需 Cookie |
| mac₂         | 基于 Responder 下发的 Cookie 重新计算                              |
| 与 C/H 链      | **无关**；MAC 不参与 Transcript（KE.md L3）                       |


MAC1 绑定 Responder 静态公钥（CONTEXT **MAC1**），用于在未完成昂贵 KEM 运算前过滤伪造握手。

---



## 7. 握手完成 → 数据面

双方派生 TK_i / TK_r 后：

- 报文类型切换为 **MESSAGE_DATA**（type=5，CONTEXT **MESSAGE_DATA**）
- 内层 IP 包经 SM4-GCM + TK 加密后在 UDP 上传输
- 重放保护、keepalive、rekey 由定时器模块独立管理

数据面细节见 [05](05-内核实现路径.md)。

## 9. 答辩 30 秒版（可背诵）

> PQ-SecTunnel 采用 Noise_IKpsk1kem 三次握手：Initiator 发 Initiation（E_i、ct₁），Responder 回 Response（ct₂、ct₃），Initiator 再发 Key Confirmation 显式确认。Static KEM（512）负责 ct₁/ct₃ 身份封装，Ephemeral KEM（768）负责 E_i/ct₂ 临时封装；Hash/KDF 用 SM3，AEAD 用 SM4-GCM。主套件 Initiation 2068 字节，需应用层分片。MAC 在 receive 入口先验，不参与 C/H 链。
