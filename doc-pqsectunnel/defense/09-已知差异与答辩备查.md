## 1. 已核对：代码 vs KE.md

| # | 议题 | KE.md / 规范表述 | 代码现状（noise.c / send.c / receive.c） | 答辩说法 |
|---|------|-----------------|----------------------------------------|---------|
| 1 | **静态 KEM 混链** | L109：ct₁/ct₃ 用 Hash(ct) 混链 | `noise_hash(ss, ct)` 四处覆写 shk 后 `mix_kem_ss`；ct₂ 直接用 shk₂ | **刻意行为**；与 ct₂ 规则不一致，未来对齐为 BREAKING |
| 2 | **H₁ 绑定 S_r^pub** | Hash(H₀‖S_r^pub) | `handshake_init()` → `mix_hash(hash, remote_static, 800)` 用**原始公钥字节** | 与 KE 公式一致；实现用 raw bytes 非 Hash(pk) |
| 3 | **mac 执行顺序** | 表列 mac 在组包序列中 | `receive.c` L135 **先** `wg_cookie_validate_packet`，**再** `consume_*` | KE 表是发送侧 transcript 顺序；接收先验 MAC，**不矛盾** |
| 4 | **应用层分片 send** | MAC 后分片 | `send.c`：`create_*` → `wg_cookie_add_mac_to_packet` → `pqc_fragment_send` | **已接入**；旧文档称未接已过时 |
| 5 | **PSK vs MAC1** | 独立机制 | PSK：`mix_psk()`；MAC1：`cookie.c` BLAKE2s("mac1----"‖S_r^pub) | **完全独立**；MAC1 不是 PSK |
| 6 | **begin_session 时机** | KE 在 KDF(C10,∅) 后 | Initiator：`send.c` 发 Confirm **前**；Responder：收 Confirm **后** | 双方最终均从 C10 派生 TK；时机不对称但密钥一致 |
| 7 | **sid_i 分配顺序** | KE 表第 1 步 | `create_initiation` **末尾** `wg_index_hashtable_insert` | 不影响 MAC 覆盖范围；仅表顺序差异 |

## 2. 静态 KEM 混链（答辩必讲，§1 展开）

KE.md L109 脚注：

- ct₁/ct₃：`encaps` 得 shk → `noise_hash` 覆写为 Hash(ct) → `mix_kem_ss`
- ct₂：`encaps` 得 shk₂ → **直接** `mix_kem_ss`

**为何主动讲**：评委可能问「规范是否统一」——答：当前代码与 KE.md 脚注一致，是已知 technical debt，改 mix_kem_ss 直接用 raw shk 需 HANDSHAKE_SPEC_REV 升版 + 全量回归。

四处 `noise_hash` 位置见 [03](03-握手协议逐步详解.md) §11。

## 3. docs/zh 旧文档 — 不可作答辩论据

写作过程对照源码确认：**以下旧 doc 说法有误或过时**，答辩勿引用。

| 旧文档说法 | 正确说法 | 依据 |
|-----------|---------|------|
| MAC1「基于 PSK」 | MAC1 密钥独立派生 | `cookie.c` + CONTEXT |
| Ephemeral 恒为 ML-KEM-512 | 支持 768（主套件即 768 临时） | `EPHEMERAL_KEM_MLKEM768` |
| McEliece Initiation「约 1116B」 | 1104B / 1016B（McEliece 套件） | `messages.h` |
| `pqc_fragment_send` 未接 send.c | **已接**（`#if PQST_APP_LAYER_FRAGMENTATION`） | `send.c` L69–70, L135–136 |
| handshake.md 密钥派生图 mix_psk 公式 | 以 KE.md 分步 KDF 为准 | `noise.c` `mix_psk` |

**答辩策略**：「项目内权威 spec 是根目录 KE.md；`doc/defense/` 与源码对齐；`docs/zh/` 为历史留档未逐一审计。」

## 4. 常见评委追问 → 一句话答

| 追问 | 答 |
|------|-----|
| 为什么 ct₁/ct₃ 和 ct₂ 混链不一样？ | KE 脚注与 noise.c 一致；历史原因，未来可能 BREAKING 对齐 |
| MAC 在 noise 里还算吗？ | 不在 noise.c；receive 入口 cookie 验，send 末尾 cookie 加 |
| 2068B 怎么过 MTU？ | 应用层 1400B 分片，MAC 后切；dmesg 可见 2 片 |
| 国密换 AES  handshake 变吗？ | 不变 ct 长度；只换 SM3/SM4-GCM 实现路径 |
| PKI 和 genkey 握手一样吗？ | 协议相同；仅 Static 钥来源不同 |
| RKEM 用了吗？ | 默认未链入；双 KEM ≠ RKEM |

更多 Q&A 见 [10-常见问题-QA.md](10-常见问题-QA.md)。
